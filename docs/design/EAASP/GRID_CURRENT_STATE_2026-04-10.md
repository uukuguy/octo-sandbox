## Executive Summary

1. **Proto v1.3 is a living, evolving contract**: 16 RPC methods defined in `runtime.proto`, and **all 16 are present in the Rust `RuntimeContract` trait** (`contract.rs:21-96`, 15 `async fn` + 1 sync `fn get_capabilities`). The three runtime implementations share the same 16-method surface but differ in internal architecture (native hooks vs monkey-patch vs SDK callbacks). [Corrected 2026-04-10: earlier claims of "13 methods" in this document were a counting error — verified all 16 methods exist in the trait.]

2. **Three L1 Runtime implementations with **divergent architectures****: Rust T1 (grid-runtime) does native hook execution; Python T1 (claude-code-runtime) uses SDK callbacks; Python T2 (hermes-runtime) monkey-patches the agent framework. They don't share code — each is a ground-up adapter.

3. **Hook execution is **not actually decoupled** from L1 Runtime**: T1 (Grid) bakes hooks into the execution loop; T2 (Hermes) intercepts tool calls at the framework level via global `_session_id` monkey-patch. HookBridge (`grid-hook-bridge`) exists but is only used by T2/T3, not production Grid.

4. **Session semantics are **fragmented****: "Session" means different things in L1 Runtime (SessionPayload, SessionHandle), L2 Skill Registry (not a concept), L3 Governance (session context for policies), and L4 Session Manager (four-plane persistence model). No unified definition.

5. **Skill Registry is a **simple REST/Git store**, not an EAASP asset manager**: Stores SKILL.md files with YAML frontmatter, versioned with git-backed commits, manually promoted through Draft→Tested→Reviewed→Production statuses. No automatic validation, no runtime type checking.

6. **L3 Governance is **policy-focused, not hook-focused****: Provides 5 API contracts (policy_deploy, intent_gateway, skill_lifecycle, telemetry_ingest, session_control) but the "hook evaluation" semantics are delegated to compiled WASM/JSON rules. The "HookBridge" is a **policy evaluation engine**, not a governance gateway.

7. **Session persistence is **not decoupled from runtime lifecycle****: Sessions are created in L1 Runtime memory. L4 Session Manager can serialize/restore them, but there's no persistent session store at L1 level — sessions die when the process dies. Pause/Resume exist in the protocol but may not be fully implemented.

8. **MCP management is **integrated into grid-engine**, not exposed as an L2 asset**: McpManager lives in `crates/grid-engine/src/mcp/` (12 modules). Connections are per-session, not shared across sessions. `eaasp-mcp-orchestrator` (L2) is a **configuration service**, not the runtime MCP manager.

9. **Memory is **multi-layered but not unified**: grid-engine exposes L0 (in-memory working), L1 (SQLite working), and L2 (semantic vector search with HNSW). Each layer has different query semantics. No layer-crossing transaction boundaries defined.

10. **v2.0 faces a hard choice**: Keep the current "Grid IS EAASP" design (T1 = native implementation), or move toward "EAASP is an abstraction Grid implements" (T1 = gRPC client to external harness). Current code assumes the former; the proto suggests the latter.

#### **2.2.3 Contract evolution: 12 → 16 method genealogy**

> **For v2.0 specification writers**: This is the authoritative breakdown of how the contract grew from "12 MUST methods" in EAASP spec v1.7 §6.5 to the "16-method service" in proto v1.3. When v2.0 writes "the L1 Runtime contract has X methods", use this table to pick X with eyes open. The discrepancy is **real history**, not a documentation drift.

**Source-of-truth chain:**

| Stage | Date | Document / Code | Method count | Where authoritative |
|-------|------|-----------------|--------------|---------------------|
| **A. Spec v1.7 §6.5 MUST** | pre-2026-04-06 | `EAASP-Design-Specification-v1.7.pdf` §6.5 | **12 MUST** | Specification (normative) |
| **B. Phase BD W1 contract** | 2026-04-06 | `crates/grid-runtime/src/contract.rs` (initial) | **13** = 12 MUST + `health` | KD-2 in EAASP_ROADMAP.md:69-79 |

| **D. proto v1.3 (BF)** | 2026-04-07 | same proto file, SessionPayload extended | **16** (no new RPCs, only field additions to SessionPayload) | Phase BF design notes |

| # | Method | Purpose |
|---|--------|---------|
| 1 | `Initialize` | Three-way handshake entry, accept SessionPayload from L4 |
| 2 | `Send` | User message → streaming response chunks |
| 3 | `LoadSkill` | Load workflow-skill (SKILL.md frontmatter + prose) |
| 4 | `OnToolCall` | PreToolUse hook interception point |
| 5 | `OnToolResult` | PostToolUse hook interception point |
| 6 | `OnStop` | Stop hook (force-continue with feedback) |

| 9 | `ConnectMcp` | Attach MCP servers to running session |

| 12 | `Terminate` | Clean up resources, flush async telemetry, emit SessionEnd |

**The 4 extension methods** (each with its rationale):

| # | Method | Stage added | Spec status | Rationale (from KD-2) |
|---|--------|-------------|-------------|-----------------------|
| 13 | `Health` | B (Phase BD W1) | **Not in spec v1.7 §6.5** | "Required for deployment/operations health checks. Reflects 'can accept new session', not just provider ping." (KD-4 in EAASP_ROADMAP.md:100). Treated as **operational extension**, not part of the 12 MUST. |
| 14 | `DisconnectMcp` | C (proto v1.2, Phase BD W3) | Not in spec v1.7 | "Symmetric counterpart to ConnectMcp. Reasonable addition." (KD-2). Lets L4 detach MCP servers without terminating the session. |
| 15 | `PauseSession` | C (proto v1.2, Phase BD W3) | Not in spec v1.7 §6.5, but **spec §8.5 has session control REST contracts (POST/GET/DELETE)** that imply pause/resume semantics | "Spec §8.5 has session control contracts, pause/resume semantics are reasonable." (KD-2). Persists state, releases compute resources, keeps session_id reservable. |
| 16 | `ResumeSession` | C (proto v1.2, Phase BD W3) | Same as PauseSession | Counterpart to PauseSession. Reconstructs runtime state from persisted snapshot. |

**The "12 MUST + 4 OPTIONAL" interpretation** (from KD-2 explicit decision, EAASP_ROADMAP.md:79):

> **Decision**: proto v1.2 may add these three methods [DisconnectMcp/PauseSession/ResumeSession] as **optional extensions** (methods exist but are non-MUST), without violating the 12 MUST core of the spec. `Health` had already been added in stage B for operational reasons.

So when v2.0 writes "the contract has N methods", the precise statements are:

- **N = 12** if you mean "what spec v1.7 §6.5 requires" → use phrase **"12 MUST methods"**
- **N = 13** if you mean "what spec v1.7 requires + Health (operational)" → rarely useful
- **N = 16** if you mean "what proto v1.3 actually defines and Rust trait implements today" → use phrase **"16-method gRPC surface"** or **"12 MUST + 4 optional (Health/DisconnectMcp/PauseSession/ResumeSession)"**

**v2.0 writing recommendation**: Pick **one canonical phrasing** for the whole spec. The cleanest is:
> "The L1 Runtime contract defines **12 MUST methods** (compliance core, unchanged from v1.7 §6.5) plus **4 OPTIONAL operational/lifecycle extensions** (Health, DisconnectMcp, PauseSession, ResumeSession). Concrete proto and Rust trait implement all 16."

This way (a) compliance certifiers can test against 12, (b) everyone reading proto sees 16 and isn't confused, (c) the 4 extensions have a documented home.

### 3.5 hermes-runtime (Python T2 Harness)

**Location**: `lang/hermes-runtime-python/src/hermes_runtime/` (8 modules, ~25 KB)

**Modules**:
- `service.py` (9,752 bytes) — gRPC RuntimeService server
- `adapter.py` (5,052 bytes) — wraps hermes_agent.Agent
- `governance_plugin/__init__.py` (~100 bytes) — monkey-patch registration

**Architecture**:

`HermesAdapter` (adapter.py):
- Wraps hermes_agent.Agent (external framework, NOT grid-engine)
- `initialize(payload) → session_id`
- `send(session_id, message) → AsyncGenerator[ResponseChunk]`
- Manages thread pool for sync→async bridging (hermes uses threading, not async)

**Thread-Bridge Design** (adapter.py):
- hermes_agent.Agent is **synchronous** (uses threading)
- HermesAdapter runs agent in ThreadPoolExecutor
- Converts async/await → thread blocking

**Governance Plugin** (governance_plugin/__init__.py):
- Global state: `_hook_bridge`, `_session_id`, `_original_handle_function_call`
- Registration: `register(ctx)` called by hermes plugin system
- **Monkey-Patch**: Replaces `model_tools.handle_function_call` with custom interceptor (line 49)
- **Interceptor Logic** (lines 52-87):
  - Pre-tool-call: queries HookBridge.evaluate_pre_tool_call
  - Decision:
    - "deny" → return error JSON
    - "modify" → update function_args
    - "allow" → proceed with original call

**⚠️ Critical Finding**: Global `_session_id` variable (line 12) is **not thread-safe**. In concurrent sessions, the `set_session_id(sid)` call (line 16) can race. This is a **known bug** (race condition in multi-session Hermes runtime).

**Hook Execution Path** (T2 vs T1 vs Grid):
- Grid (T1 Rust): Hooks fire at HookRegistry.execute() → synchronous, in-engine
- Claude-Code (T1 Python): Hooks fire at HookExecutor callbacks → calls external HookBridge REST
- Hermes (T2 Python): Hooks fire via model_tools.handle_function_call monkey-patch → interceptor checks HookBridge

**⚠️ Finding**: Three different hook paths with no unified semantics. The proto spec doesn't enforce which one is "correct."

- No mock for hermes_agent (tests assume it's available)

### 10.3 Session Semantics Divergence

**"Session" means different things in different layers**:

1. **L1 Runtime Level** (grid-runtime/contract.rs:101-134):
   - SessionPayload: initialization input {user_id, quotas, hooks_json, ...}
   - SessionHandle: {session_id} opaque reference
   - Execution semantics: message → response stream

2. **L2 Skill Registry** (no session concept):
   - Skills are global assets, not session-scoped
   - Versioning per skill, not per session

3. **L3 Governance Level** (eaasp-governance/models/):
   - Session = context for policy evaluation {user, org_unit, session_id}
   - Policies are session-scoped
   - Reference: policy.scope = "session"

4. **L4 Session Manager Level** (eaasp-session/models.py):
   - Session = persistent entity {user_id, runtime_id, state, created_at}
   - Session = coordination unit for pause/resume/terminate
   - Reference: models.py SessionData

**Timeline**:
```
L4: Create Session → assign to L1 Runtime
L1: Initialize → create SessionPayload + SessionHandle
L3: Policy Engine → receives session context for decision-making
L1: Send/Terminate
L4: Update Session state → persist snapshot
```

**Consistency Issues**:
- L1 SessionPayload.hook_bridge_url is L3-provided, but L1 Rust T1 ignores it
- L4 Session.state is opaque; if L1 adds fields, L4 can't parse old snapshots
- Session pause/resume is defined in proto but may not be implemented (grid-runtime impl unclear)

### 10.4 Hook Execution Path Comparison (T1 vs T2)

#### **Grid T1 (Rust)**

```
Agent Loop (agent/loop_.rs)
  ↓ [PreToolUse event]
  ↓ HookRegistry.execute(HookPoint::PreToolUse, context)
  ↓ [for each hook in registry]
  ↓ HookHandler.handle(context) → HookAction
  ↓ [Blocks agent loop until complete]
  ↓ Decision: Allow/Deny/Modify
  ↓ [Continue execution or abort]
```

**File**: crates/grid-engine/src/hooks/registry.rs (sync execution)
**Timing**: Synchronous, blocking
**Scope**: In-process
**Decision**: HookAction enum (Continue, Abort, Modify)

#### **Claude-Code T1 (Python)**

```
Service.on_tool_call(ToolCallEvent)
  ↓ HookExecutor.on_pre_tool_call(tool_name, input)
  ↓ [if hook_bridge configured]
  ↓ HTTP POST to hook_bridge_url/evaluate
  ↓ [Awaits response]
  ↓ Decision: "allow" | "deny" | "modify" (string, not enum)
  ↓ [Return HookDecision proto]
```

**File**: lang/claude-code-runtime-python/src/claude_code_runtime/hook_executor.py:40-80
**Timing**: Asynchronous (async def), awaited
**Scope**: Out-of-process (REST to external HookBridge)
**Decision**: String-based ("allow" | "deny" | "modify")

#### **Hermes T2 (Python)**

```
HermesAgent.send()
  ↓ [Agent calls model_tools.handle_function_call(func_name, args)]
  ↓ [Monkey-patched interceptor activates]
  ↓ Interceptor._intercepted_handle_function_call()
    ├─ HookBridgeClient.evaluate_pre_tool_call(session_id, tool_name, input)
    ├─ [Checks decision = "allow" | "deny" | "modify"]
    ├─ if deny: return error JSON
    ├─ if modify: update function_args
    └─ Call _original_handle_function_call(...)
  ↓ [Execution continues or errors]
```

**File**: lang/hermes-runtime-python/src/hermes_runtime/governance_plugin/__init__.py:44-89
**Timing**: Synchronous (inline interceptor)
**Scope**: In-process interceptor, but calls out-of-process HookBridge
**Decision**: String-based ("allow" | "deny" | "modify")

**Decision Format Mismatch**:
- Grid: HookAction enum
- Claude-Code: String-based ("allow", "deny", "modify")
- Hermes: String-based, same as Claude-Code

**⚠️ Finding**: Decision semantics are not consistent across implementations. Grid's HookAction (Continue, Abort, Modify) doesn't map 1-1 to string-based decisions. This would be a breaking change in v2.0 if unified.

### 10.6 Known Bugs and Technical Debt Surfaced by Reading Code

#### **Bug 1: Global Session ID Race Condition in Hermes T2** 

**File**: lang/hermes-runtime-python/src/hermes_runtime/governance_plugin/__init__.py:12

```python
_session_id = ""  # set per-session by hermes-runtime adapter

def set_session_id(sid: str):
    global _session_id
    _session_id = sid
```

**Problem**: In concurrent sessions, `set_session_id()` can race. If two sessions are active simultaneously:
1. Session A calls `set_session_id("s-1")`
2. Thread switches to Session B, calls `set_session_id("s-2")`
3. Thread switches back to A, but `_session_id = "s-2"` (wrong)
4. A's hook evaluation uses B's session context

**Impact**: HIGH. Governance decisions leak across sessions.

**Mitigation**: Use thread-local storage or async-safe session context (e.g., contextvars).

## 11. Relevance to EAASP v2.0 Design Decisions

### **Q1: Is L1 Harness Actually Decoupled from Sandbox?**

**Answer**: **Partially no.**

**Evidence**:
- Grid-runtime.harness hardcodes sandbox_id = "default" (harness.rs:134)
- No way to specify which sandbox to use at the harness level
- Sandbox selection happens in grid-engine.AgentRuntime, not grid-runtime.harness
- To swap sandbox (Docker → WASM), you modify grid-engine, not grid-runtime or grid-sandbox

**Implication for v2.0**: If you want Harness/Sandbox decoupling, you need to:
1. Accept sandbox_id as a parameter in RuntimeContract.initialize()
2. Move sandbox selection logic into harness (not engine)
3. Or: define separate Sandbox RPC service, decouple from RuntimeService

**Current Status**: Harness is **bound to grid-engine's default sandbox choices**. Not truly decoupled.

---

### **Q2: Is Session Persisted Independently from Container Lifecycle?**

**Answer**: **No, with caveats.**

**Evidence**:
- L1 Runtime sessions live in memory (DashMap in AgentRuntime)
- Session persistence is **optional** via SqliteSessionStore (only if enabled)
- If L1 Runtime process dies, **all in-memory sessions are lost** (unless they were flushed to SQLite)
- Pause/Resume exist in proto but may not flush to disk immediately
- L4 Session Manager can serialize/restore, but L1 Runtime doesn't automatically persist on pause

**Implication for v2.0**: If you want persistent sessions independent of runtime lifecycle:
1. Make SqliteSessionStore mandatory, not optional
2. Flush on every state change, not just on terminate
3. Or: define separate Session Store service (L4 owned, not L1)

**Current Status**: Sessions **die with the process** unless explicitly persisted to L4. No independent persistence at L1 level.

---

### **Q3: Is "Environment" Currently a First-Class Concept?**

**Answer**: **No.**

**Evidence**:
- No "Environment" type in grid-types
- SandboxProfile (Dev/Stg/Prod) exists but is not a first-class entity
- Sandbox is selected by grid-engine internally, not exposed as a user-facing concept
- Proto doesn't define Environment as a top-level concept

**Implication for v2.0**: Environment would need to be:
1. A first-class concept in proto (SessionPayload.environment_id?)
2. Mapped to SandboxProfile + SecurityPolicy + ResourceQuotas
3. L1 Runtime must accept it and configure sandbox accordingly

**Current Status**: Environment is **implicit** (baked into sandbox selection), not explicit.

---

### **Q4: Does Current L1 Runtime Support "Multiple Harnesses per Sandbox" or "Multiple Sandboxes per Harness"?**

**Answer**: **Neither clearly, Grid does multiple sessions per harness + single default sandbox.**

**Evidence**:
- Grid-runtime can manage multiple sessions (via sessions DashMap)
- But all sessions use sandbox_id = "default" (hardcoded in initialize)
- Cannot assign different sandboxes to different sessions
- Cannot have two harnesses share one sandbox (no inter-process sandbox sharing defined)

**Implication for v2.0**: 
- Multi-harness per sandbox: Would require sandbox to be a first-class RPC service (not tied to harness)
- Multi-sandbox per harness: Would require sandbox selection to be per-session (not hardcoded)

**Current Status**: Grid is **1-to-many harness-to-sessions, 1-to-1 harness-to-sandbox (default)**. Other combinations not supported.

---

### **Q5: Where Are Credentials Currently Stored? Can They Leak into Sandbox?**

**Answer**: **Credentials are in provider config + environment variables. Leakage is plausible but not proven by this audit — requires targeted verification before writing into v2.0 as a P0 security gap.**

**Evidence (partial — verification needed)**:
- API keys: Stored in ProviderConfig (crates/grid-types/src/provider.rs) ✓ verified
- Resolution: Uses `resolve_api_key_env()` from providers module (reads $ANTHROPIC_API_KEY, etc.) ✓ verified
- ⚠️ **NOT verified**: Whether DockerAdapter/SubprocessAdapter filter parent env before spawning sandbox. Audit noted "Application code sanitizes env before spawning (unclear from code read)" — this is the exact point that determines P0 vs non-issue.
- ⚠️ **NOT verified**: Whether StdioMcpClient passes parent env to MCP server subprocesses.

**Action for v2.0 writing**: Before citing this as a security gap, do a targeted read of:
- `crates/grid-sandbox/src/docker.rs` (env handling in Bollard HostConfig)
- `crates/grid-engine/src/mcp/client/stdio.rs` (env handling in tokio::process::Command)

**Risk Assessment**:
- HIGH: If agent has shell/bash tool, it can `env` to see API keys
- MEDIUM: MCP servers spawn as child processes, inherit parent env
- LOW: Application code sanitizes env before spawning (unclear from code read)

**Implication for v2.0**: Need explicit credential isolation:
1. Sandbox should NOT inherit parent environment (use allowlist)
2. Credentials should be passed as file descriptors or IPC, not env vars
3. MCP servers should receive only whitelisted env vars

**Current Status**: **Credentials can leak to sandboxed agents**. No explicit isolation.

### **Q6: Is Current Hook Execution Path Appropriate as "Agent Gateway"?**

**Answer**: **No, it's more of an "Agent Interceptor".**

**Evidence**:
- Grid hooks are **in-process** (can't enforce access control across trust boundaries)
- Decision format is not standardized (enum vs string, inconsistent across T1/T2)
- Hook execution is synchronous (blocks agent loop, no timeout)
- Hermes uses monkey-patch (bypasses normal control flow, hard to audit)
- Claude-Code uses callbacks (async, but can fail silently)

**Gateway Characteristics** (should have):
- Centralized policy enforcement
- Standardized decision format
- Timeout + escalation on policy engine failure
- Audit trail of all decisions

**Current Implementation**:
- Decentralized (each harness has hooks)
- Non-standardized decisions (breaks compatibility)
- No timeout protection
- Audit trail optional (depends on telemetry collection)

**Implication for v2.0**: If you want "Agent Gateway" (v2.0 mention), you need:
1. Move hooks out of harness → external service (like HookBridge)
2. Standardize decision format (only proto, not strings)
3. Add timeout + circuit-breaker
4. Mandate audit logging

**Current Status**: Hook execution is **not appropriate as a gateway**. It's too decentralized and inconsistent.

### **Q7: What's Current Granularity of Telemetry? Can We Distinguish Business vs AI Quality Metrics?**

**Answer**: **Granularity exists, but distinction is not explicit.**

**Evidence**:

TelemetryEvent fields (common.proto:41-49):
- session_id, runtime_id, user_id
- event_type: "tool_call" | "tool_result" | "hook_decision" | "session_end" | ...
- timestamp, payload_json
- resource_usage: { input_tokens, output_tokens, compute_ms }

**Business Metrics** (implicitly captured):
- User engagement (session_id → user behavior)
- Cost (resource_usage.input_tokens + output_tokens)
- Latency (compute_ms)

**AI Quality Metrics** (NOT explicitly captured):
- Accuracy (would need ground-truth comparison in payload_json)
- Hallucination detection (would need semantic analysis)
- Token efficiency (output_tokens / usefulness score)

**TelemetryEvent.payload_json** is opaque — could contain anything, but no schema enforces quality metrics.

**Implication for v2.0**: To distinguish metrics:
1. Define structured payload schema per event_type
2. business_metrics: cost, latency, user_id, outcome (success/failure)
3. ai_metrics: model confidence, reasoning steps, external validation
4. Mandate that both are collected (currently optional)

**Current Status**: **Telemetry granularity is high, but distinction is implicit and optional**. No enforced schema.

### **Q8: How Much of the "7-Layer Control Plane" is Actually Present in Code?**

**Answer**: **2.5/7 layers are implemented. Layers are not clearly delineated.**

**Evidence**:

Traditional 7-layer model (if applied to EAASP):
1. **Physical**: Firecracker/Docker containers → Implemented (Docker only, no Firecracker)
2. **Link**: Inter-process communication (gRPC, REST) → Implemented (gRPC for Runtime, REST for L2/L3/L4)
3. **Network**: VPC, isolation, firewall rules → Partially (SandboxProfile.Dev/Stg/Prod, but not enforced)
4. **Transport**: Protocol handlers → Implemented (tokio async runtime)
5. **Session**: Session management → Partially (L1 runtime sessions + L4 orchestration, but different models)
6. **Presentation**: User-facing APIs → Implemented (SDK, CLI, HTTP API)
7. **Application**: Agent logic → Implemented (AgentLoop, Agent executor)

**EAASP 7-Layer Control Plane** (if reinterpreted as EAASP):
1. **L1 Runtime Control**: Agent execution, hooks → Implemented
2. **L2 Asset Control**: Skill registry, MCP orchestrator → Implemented (but simple)
3. **L3 Governance Control**: Policy enforcement, intent routing → Partially (policy deployment works, but enforcement unclear)
4. **L4 Orchestration Control**: Session routing, failover → Partially (basic orchestration, no failover)
5. **Security Control**: Auth, secrets, audit → Partially (auth.rs exists, but incomplete)
6. **Observability Control**: Telemetry, logging, tracing → Partially (telemetry works, tracing configured)
7. **Data Control**: Memory, state, persistence → Partially (multi-layer memory, but no unified persistence)

**Implication for v2.0**: Current "control plane" is **weak and non-hierarchical**. To implement true 7-layer control:
1. Define each layer's scope clearly (not overlapping)
2. Make layer boundaries explicit in code (not implicit)
3. Enforce data flow through layers (not shortcuts)

**Current Status**: Code is **organized by function (runtime, governance, orchestration), not by control layers**. Layers are implicit, not enforced.

## 12. Open Questions for v2.0 Writing

1. **Session Persistence Responsibility**: Should L1 Runtime be responsible for persisting sessions, or should that be L4's job? Currently, both can do it, leading to ambiguity.

2. **Hook Decision Format Unification**: Should v2.0 standardize on enum (Rust) or string ("allow"/"deny"/"modify")? This is a breaking change.

3. **Sandbox-as-a-Service**: Should sandbox selection be a separate RPC service, or remain embedded in harness? Current design embeds it, but multi-harness-per-sandbox would require decoupling.

4. **Credential Isolation**: How should secrets be passed to sandboxes without environment variable leakage? Current design leaks.

5. **Hook Streaming vs One-Shot**: Proto defines bidirectional streaming for HookBridge, but implementations prefer one-shot EvaluateHook. Which is the canonical pattern?

6. **Environment as First-Class Concept**: Should v2.0 expose "Environment" (Dev/Stg/Prod/Custom) as a SessionPayload field, or is SandboxProfile sufficient?

7. **Multi-Session Harness Deployment Model**: Does a single harness process manage multiple sessions (current), or should each session get its own harness process (per_session mode)? Proto allows both, but implications differ.

8. **Pause/Resume Completeness**: Are pause/resume fully implemented in production, or are they proto-only placeholders? Current code suggests the latter.

## Conclusion

This audit reveals that **Grid is a production-grade EAASP v1 implementation**, but the current architecture has several impedances with the EAASP spec:

1. **Proto-Trait Mismatch**: Rust trait doesn't enforce all 16 methods; gRPC service does
2. **Hook Inconsistency**: Three different hook execution paths across T1/T2 implementations
3. **Session Overloading**: Same term means different things at L1/L3/L4
4. **Missing Enforcement**: Quotas, skill search scope, managed hooks all ignored
5. **Decoupling Issues**: Harness tied to grid-engine; sandbox not truly pluggable

**For v2.0**, the design should:
- Clarify whether EAASP is "Grid's protocol" (current) or "Grid's interface" (aspirational)
- Unify hook semantics across all implementations
- Make session persistence explicit and mandatory
- Expose sandbox/environment as first-class concepts
- Move toward a "gateway" architecture (external hook evaluation) instead of in-process

The codebase is well-engineered and production-ready for the Grid use case, but would benefit from tightening the EAASP abstraction boundaries before scaling to multiple implementations.
