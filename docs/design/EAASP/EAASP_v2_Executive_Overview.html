01 — Origins

Why EAASP Exists

Enterprise AI has reached an inflection point. Large language models can reason, plan, and execute complex tasks — but deploying them at enterprise scale exposes a critical gap between model intelligence and organizational readiness. Industry surveys show that while 88% of organizations use some form of AI, fewer than 10% have successfully deployed multi-step agentic systems into production business processes.

The barriers are not intelligence — they are governance, security, operational control, and the ability to integrate heterogeneous agent engines into a coherent platform. Enterprises need answers to questions that no single agent framework addresses: How do we enforce organizational policy across all agents uniformly? How do we ensure every agent conclusion is traceable to evidence? How do we let business analysts create intelligent workflows without writing code? How do we manage credentials, data sovereignty, and audit trails at scale?

EAASP was created to answer these questions. It is not an agent — it is the platform that hosts, governs, and orchestrates agents. It treats agent runtimes the way an operating system treats applications: providing the execution environment, enforcing security boundaries, managing shared resources, and presenting a unified interface to users — regardless of which agent engine runs underneath.

The Managed Agents Convergence

In April 2026, Anthropic released Managed Agents — a hosted service for running long-horizon autonomous agents. The accompanying engineering blog articulated a profound architectural insight: decouple the brain from the hands from the session. Agent components (the reasoning harness, the execution sandbox, the durable session log) should be independent interfaces, not a monolithic container. This mirrors how operating systems virtualized hardware into abstractions general enough for programs that did not exist yet.

EAASP v2.0 absorbs the most valuable ideas from Managed Agents — the append-only session event stream, structural credential isolation, and the meta-harness philosophy — while preserving what Managed Agents does not provide: enterprise governance, multi-runtime heterogeneity, organizational access control, evidence chains, and the complete collaboration lifecycle from event detection through memory deposit.

The core relationship: Managed Agents is infrastructure-level abstraction (how to run one agent reliably). EAASP is platform-level architecture (how to deploy, govern, and orchestrate many agents across an enterprise). They are complementary strata — you stack them, you don't choose between them.

02 — Goals

What EAASP Achieves

⚡

Runtime-Agnostic Execution

Any agent engine — Claude Code, Aider, Goose, LangGraph, CrewAI, or custom-built — can join the execution pool by implementing a standard runtime contract. The platform selects the best runtime per task. Enterprises are never locked to a single vendor or framework.

🛡

Unified Agent Governance

A single set of managed hooks applies to every agent identically. Administrators deploy a policy once; the platform compiles it into deterministic enforcement that runs inline in every runtime. Deny always wins — no agent can override organizational policy.

📡

Event-Driven Proactive Service

Agents don't wait for user commands. The event engine aggregates raw alerts, logs, and change events into structured business-event objects. One hundred alerts become a single actionable event. Agents shift from passive order-takers to proactive scene-watchers.

🔗

Evidence Chains & Audit

Every agent conclusion must reference evidence anchors — immutable snapshots of the original data. Auditing moves from "recording what happened" to "proving why this judgment was made." Full traceability from conclusion to source data.

🧠

Persistent Memory as an Asset

Memory is not session-ephemeral state. The Memory Engine provides three-layer persistent storage — evidence anchors, file-based memory, and hybrid retrieval — enabling agents to accumulate experience, remember preferences, and improve over time.

🏭

Agent Factory for the Organization

Business analysts create intelligent workflows through a visual process designer — no code required. Successful interactions are extracted into reusable skills. A continuous quality loop ensures the platform improves with every execution.

04 — Design Philosophy

Core Design Principles

Every architectural decision in EAASP is tested against these principles. They emerged from four iterations of design evolution and were sharpened during the v2.0 analysis against Managed Agents' meta-harness philosophy.

01

The platform provides governance; the runtime provides intelligence.

The master boundary principle. The platform defines what must happen (hooks, policies, approval chains, credential scoping). The runtime decides how to accomplish the work (planning, tool selection, error recovery). Neither crosses into the other's domain.

02

Deny always wins.

If any hook at any scope level denies an action, the action is blocked — regardless of what other hooks or the agent's reasoning concludes. This is deterministic, not probabilistic. It is the guarantee that makes hooks suitable for enterprise governance.

03

Hooks provide guarantees; agents provide judgment.

Things that must happen every time (policy enforcement, audit, quality checks) are hooks. Things requiring adaptation and reasoning (plan decomposition, creative problem-solving) are agent intelligence. Do not substitute one for the other.

04

Governance is permanent; governance configurations are not.

The hook protocol, approval chain mechanism, and audit pipeline are stable architectural elements. The specific thresholds, policies, and rules are living configurations that must evolve as models improve. Assumption debt — stale configurations calibrated to older models — is actively managed through observability, shadow testing, and graduated relaxation.

05

Drafts first — never overreach.

Agents produce drafts, never directly executing high-risk operations. All write operations pass through the five-stage approval control chain: Plan → Check → Draft → Approve → Execute. The cost of a draft is seconds; the cost of overreach may be irreversible.

06

Model-agnostic by configuration.

No architectural component depends on a specific inference provider. Model backend selection is an environment variable. The same platform runs on Claude, Llama, Mistral, or any capable model — enabling fully private deployment without architectural changes.

07

Extend, never rebuild.

Each evolution phase adds components incrementally without modifying existing capabilities. The v2.0 upgrade demonstrates this: six enhancements added, zero v1.8 components removed or restructured.

06 — Key Capabilities

Platform Capabilities in Detail

Heterogeneous Runtime Pool

L1 hosts any agent engine that implements the standard runtime contract: 12 MUST methods (the compliance core) plus 4 optional operational extensions. Runtimes are organized in four tiers by adapter thickness:

TierAdapterExamplesHook Integration
T0Managed harness (external)Anthropic Managed AgentsSSE stream interception by adapter
T1Thin (native compatibility)Claude Code, grid-engine, Agent SDKNative hooks — zero overhead
T2Medium (partial compat.)Aider, Goose, Roo Code CLI, Cline CLIHook bridge sidecar
T3Thick (full translation)LangGraph, CrewAI, Pydantic AIHook bridge + skill translation

Five selection strategies (task-matched, blind-box, user preference, A/B testing, cost-optimized) route each session to the best available runtime. The blind-box strategy generates empirical quality data by presenting two runtimes' outputs side-by-side without revealing which produced which.

Four-Card Interaction Paradigm

L5 replaces traditional chat with structured decision support. When a business event occurs, users see four pinned cards covering the complete decision cycle: the Event Card (what happened, how severe), the Evidence Pack (data snapshots with anchor IDs), the Action Card (step tree with preconditions), and the Approval Card (scope, rationale, rollback path, approve/reject buttons). Each card enables a decision within 30 seconds.

Approval Control Chain

L3 provides a five-stage governed execution chain: Plan (agent forms a read-only operation plan) → Check (deterministic verification via rule engine or simulation — same input always produces same output) → Draft (agent produces an executable draft but does not execute) → Approve (human authorization via 30-second approval card) → Execute (actual execution via MCP). Risk tiers determine which stages are required: low-risk operations skip all stages; high-risk traverse the full chain.

Session Event Stream (v2.0)

Every session writes to an append-only event stream capturing the complete, ordered sequence of events: messages, tool calls, hook decisions, compaction points, and memory operations. This enables crash recovery (replay from last event), reversible compaction (compacted messages remain accessible), and complete audit replay. The event stream is the substrate; audit records, memory, and checkpoints are derived views.

Tiered Credential Isolation (v2.0)

MCP credentials are structurally isolated using a mechanism matched to each runtime tier. T0 sessions use a full credential proxy — the sandbox never sees credentials. T1 runtimes receive pre-bundled ephemeral tokens (per-session, least-privilege, auto-expiring). T2/T3 runtimes use hook-bridge-injected credentials — the bridge strips placeholders, injects real credentials from the vault per-call, and strips them from responses. The security outcome is equivalent across all tiers; the mechanism respects runtime autonomy.

Assumption Lifecycle Management (v2.0)

Hooks, skills, and policies encode assumptions about model capabilities that go stale as models improve. Left unmanaged, this creates assumption debt — accumulated governance configurations that add latency and friction without improving outcomes. The platform provides: assumption observability (tracking hook effectiveness via activity-outcome correlation), proactive staleness detection (surfacing low-impact hooks and over-restrictive policies), and controlled testing (hook shadow mode, A/B skill testing, graduated policy relaxation).

09 — Security & Governance

Enterprise Security Architecture

EAASP's security model operates on the principle that the most dangerous threat is not external attack but an overly capable internal agent acting without sufficient constraints. Three layers of defense:

🔒

Hook-Based Policy Enforcement

14 lifecycle hook events across three layers. Four handler types (command: ms, http: ~300ms, prompt: ~2s, agent: ~15s). OPA/Rego as a first-class policy backend for microsecond deterministic evaluation. Deny always wins across all scope levels.

🏗

Sandbox Isolation Tiers

Configurable per organizational unit: Standard (Docker), Kernel (gVisor — syscall interception), Hardware (Firecracker/Kata — full VM boundary). All tiers enforce ephemeral execution — containers created per-session, destroyed on completion.

🌐

Egress Control & DLP

Governed policy type: domain allowlists (network-level), content inspection (PreToolUse hooks examining outbound data for PII and classified markers). Blocks unauthorized destinations and unauthorized data on authorized destinations.

Data Tier

PostgreSQL for user profiles, policy configuration, skill metadata. TimescaleDB/ClickHouse for audit event store and execution logs. Redis for session state checkpoints (AOF persistence, cluster mode). Session Event Stream via Kafka/NATS JetStream or append-only S3. Object storage for skill content and output files. Secrets vault for all credentials (rotated automatically, never in config files).

Network Security

Three zones: public (CDN + API gateway), internal (L4/L3/L2 via service mesh with mTLS), runtime (L1 with restricted egress — can reach MCP servers, L2 skill repo, L3 audit, and Session Event Stream only). A compromised agent cannot access user databases, admin interfaces, or other agent sessions.

11 — Roadmap

Development Roadmap

The platform evolves through seven phases. Each delivers independent value with no rework of prior phases. v2.0 capabilities integrate into existing phases rather than adding new ones.

Phase 0

MVP Validation

Full L4→L3→L2→L1 chain. Policy compiler, five API contracts, three-way handshake, skill injection, hook enforcement. Session Event Stream provisioning. Validation: a complete workflow-skill executes end-to-end under managed-hooks governance.

Phase 1

Event-Driven Foundation (Weeks 1–6)

L4 event engine (ingest → dedup → cluster → event objects → state machine). L5 event room data model and event-card API. L4 hooks (EventReceived, PreSessionCreate, PostSessionEnd).

Phase 2

Memory & Evidence (Weeks 5–12)

L2 Memory Engine (evidence anchors + file memory + hybrid index). Memory write channel. Context assembly channel. Skill extraction mechanism (event-stream-to-skill). Reversible compaction via event stream.

Phase 3

Approval & Verification (Weeks 10–18)

L3 approval control chain (Plan → Check → Draft → Approve → Execute). Deterministic verifier. L5 approval card. OPA/Rego policy backend. Egress control policies. Sandbox isolation tiers.

Phase 4

Multi-Agent Collaboration (Weeks 16–24)

L4 A2A router (capability publish, dispatch, aggregation). Parallel execution. L5 review-conclusion display. T0 Managed Harness as A2A-capable participant.

Phase 5

Complete Collaboration Space (Weeks 22–30)

L5 four-card rendering engine. IM bot plugins. Retrospective and memory-deposit closed loop. Assumption health dashboard. Hook shadow mode.

Phase 6

Ecosystem Expansion (Ongoing)

Runtime certification pipeline. L2 skill marketplace. Multi-tenancy. Platform API SDK. Assumption lifecycle management process. Hook effectiveness analytics. Graduated policy relaxation. Tiered credential isolation.
