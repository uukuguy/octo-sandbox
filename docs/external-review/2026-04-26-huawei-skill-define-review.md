## 二、问题诊断 — 为什么不精简?

### 1. 目录爆炸

第 1 部分通用结构已经 9 个目录,第 2 部分电力专属又是 10 个目录(`01_` 到 `10_`),还各有 2-4 个子文件。看起来"全",但 80% 的项目用不到一半。

### 2. "独有要素"5 选 2 注水

实际真正"电力调度特有"的:

| 真特点(应高亮) | 名义"独有"但其实通用 |
|---------------|------------------|
| **机理 + AI 双引擎,机理硬约束不可被 AI 覆盖** ⭐ | 调度层级元数据(只是 tag 字段) |
| **三重校核 pre/process/post + 全维度物理量校验**(N-1/潮流/限额)⭐ | 强可解释输出(任何审计场景都该有) |
| **闭锁逻辑内嵌 + 操作票输出**(直接对接现场)⭐ | 权限分级(任何 B2B 系统都有) |
| | 闭环评估迭代(几乎所有 ML 系统都有) |
| | 异常降级(基本工程标准) |

### 3. 同一概念分散

"三重校核"(pre/process/post)分别在第 06 目录、独有要素 (3)、对照表"校核层"出现 3 次,内容重复。

### 4. 结尾推销味

"需要我给你一份可直接复制粘贴的 yaml 元数据模板 + 输入输出 schema 样例吗?直接就能落地写代码" 是 GPT/Claude 应答模板的尾巴,不应该出现在标准化方案里。

### 5. 文风过于"概念展示"而非"约束规范"

列出"应该有什么",**没说**:
- "不应该有什么"(反向边界)
- "边界在哪里"
- "谁来强制"
- "怎么校验"

规范的本质是**约束 + 可执行性**,这份方案更像目录模板介绍。

## 三、给领导的 1 段话回复(可直接转发)

> "原方案结构完整,但**不够精简,且通用要素与电力专属要素混在一起**。真正特点突出的只有 3 条:**机理+AI 双引擎(机理硬约束不可被 AI 覆盖)、三重物理校核(pre/process/post + N-1/潮流/限额校验)、闭锁逻辑内嵌**。建议设计的规范应分两层:**L1 通用层直接引用 MCP/JSON Schema/SemVer 等业界工业标准**(不重复造),**L2 调度行业层只规定上述 5 条 MANDATORY**(双引擎 / 三重校核 / 强可解释 / 闭锁 / 时效分级)。规范应使用 MUST/SHOULD/MAY 三级 + 每条带可执行校验,而不是大目录模板。我可以起一个 ~3 页的草案,2 天内出。"

## 四、新规范设计方向(草案讨论用)

> 详细草案待与领导对齐后展开。本节只列设计原则,不出完整规范。

### 核心思路: 三层切分

```
L1 通用 Skill 标准 —— agent industry baseline (跨行业)
   ↓
L2 行业 Profile —— 电力调度行业差异层(本规范的核心价值)
   ↓
L3 项目实例 —— 某省调 / 某局的具体落地
```

原方案把 L1 + L2 混在一起讲,所以看起来又"全"又"乱"。新规范应该 L1 直接引业界标(MCP / JSON Schema / SemVer / JSON-RPC 2.0 错误模型),不重写轮子;L2 只规定行业差异。

### 草案骨架(~3 页,6 节)

```
§1 适用范围 —— L1 / L2 分层声明
§2 L1 通用层 —— 直接引用业界标(MCP / JSON Schema / SemVer)
§3 L2 调度行业 MANDATORY 5 条:
   §3.1 机理 + AI 双引擎硬约束
   §3.2 三重校核 pre/process/post
   §3.3 输出可解释性(规程条款引用 + 量化校核)
   §3.4 闭锁绑定(控制类 skill MANDATORY)
   §3.5 时效分级(seconds/minutes/hours/days)
§4 强制等级 MUST/SHOULD/MAY
§5 不属于本规范的事(反向边界)
§6 演进机制 —— ADR-style + 工作组评审
```

### 关键改进对比

| 维度 | 原方案 | 新规范 |
|------|--------|--------|
| 篇幅 | ~2 页厚 + 重复 | ~3 页紧凑 |
| 强制性 | 没区分 must/should/may | 每条 §3.X 标 MANDATORY |
| 通用 vs 行业 | 混在一起 | L1 引业界标 + L2 行业专属,不重写轮子 |
| 可校验 | 模糊"应当满足" | 每条都有"未满足→XXX 错误"的可执行判断 |
| 反向边界 | 没说不要做什么 | §5 显式列"不在规范内" |
| 演进机制 | 没说 | §6 ADR-style 演进 + 工作组评审 |

### 5 个 MANDATORY 草拟要点(每条加可执行校验)

#### §3.1 机理 + AI 双引擎硬约束

- skill 元数据 MUST 声明 `mechanism_constraints[]` (引用规程条款 + 数值阈值)
- AI 输出值若违反 mechanism_constraints,skill MUST reject + 返回 `MECHANISM_VIOLATION` 错误
- 测试用例 MUST 包含至少 1 个"AI 推荐方案违反 N-1"的反例,验证 reject 行为

#### §3.2 三重校核

- pre_check: 权限 + 输入合法性,在 LLM 调用前
- process_check: 潮流/稳定边界,在 LLM 输出后落盘前
- post_check: 指令合规性,在指令派发前
- 任一 check 失败 → 整体降级 / 拒绝

#### §3.3 输出可解释性

- 每条调控建议 MUST 附:
  - 引用的规程条款 ID(如 GB/T 31464-2015 §X.Y)
  - 量化校核结果(数值 + 阈值 + 余量)
  - 影响范围评估(失负荷量、稳定裕度变化)
- 缺任意一项 → 输出无效

#### §3.4 闭锁绑定(控制类 skill MANDATORY)

- 控制类 skill(发指令、改定值、合分闸)MUST 声明 `interlock_required: true`
- 调用方 MUST 提供 `interlock_token`,skill 内部 verify 通过才执行
- 无 token 的控制类 skill 调用 → `INTERLOCK_REQUIRED` 错误

#### §3.5 时效分级

- skill 元数据 MUST 声明 `time_class:` 之一: `seconds_AGC` / `minutes_powerflow` / `hours_dayplan` / `days_outage`
- 编排引擎据此调度优先级,平台据此监控 SLA

## 五、待与领导对齐的开放问题

讨论新规范草案前,需要确认以下几点(影响规范深度和篇幅):

1. **受众是谁?**
   - 选项 A: 调度领域 AI 开发者(写 skill 的人)→ 规范要给开发约束
   - 选项 B: 项目经理 / 标准化办公室 → 规范要给评审 checklist
   - 选项 C: 友商对接(如华为、南瑞)→ 规范要给互操作契约
   - 不同受众对篇幅和精度要求差异大

2. **规范是开源还是内部?**
   - 开源 → 必须避免引用具体公司内部系统名(D5000 / 调控云内部代号)
   - 内部 → 可以直接绑定具体平台

3. **跟现有标准的关系?**
   - 是替代既有标准(GB/T、电力行业标准)还是补充?
   - 是否要走标准化委员会(像 IEC TC57)流程,还是企业内部规范?
   - 这决定规范用语严谨度

4. **5 条 MANDATORY 是否需要再裁剪 / 增加?**
   - 时效分级(§3.5)是否真"行业必需"?或可降为 SHOULD?
   - 是否漏了"数据治理"(客户数据 / 实时量测溯源)层?

5. **草案落地时点?**
   - "2 天出 ~3 页草案"承诺是否合适?
   - 还是先出 1 页框架草案,再 1 周深化?
