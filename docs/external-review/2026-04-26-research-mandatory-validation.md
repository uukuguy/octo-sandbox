# Research:MANDATORY 验证 + 漏点扫描

**研究日期**: 2026-04-26
**目的**: 验证原方案 5 条 MANDATORY 是否真的最重要,**重点扫描原方案漏掉的、对国调中心电力调度场景更关键的 MANDATORY**
**研究方式**: 基于电力行业知识 + 调度业务常识做批判性判断,**未独立联网核对具体国标条款号**(下文凡引用 GB/DL/T/Q 等编号均为示例性参考,正式发布前需电力专家校对)
**站位**: 模拟电力调度生产专家给标准委员会写初步意见,不为原方案辩护,只追"什么会出生产事故"

---

## §0. 评审基准:这是"控制类 skill"标准,不是"问答类 skill"标准

在判断每一条之前,先锁定一个范围共识 —— 否则讨论会一直跑偏:

电力调度场景里,AI skill 至少分三类,**MANDATORY 强度差别巨大**:

| 类别 | 例子 | MANDATORY 强度 |
|------|------|---------------|
| **A. 控制/决策类** | 发遥控指令、改定值、合分闸、发 AGC 指令、生成操作票 | **最高 — 直接动一次电网** |
| **B. 辅助分析类** | 潮流计算辅助、检修计划生成、负荷预测、新能源消纳分析 | 中 — 错了影响调度员判断,但有人复核兜底 |
| **C. 信息查询类** | 规程检索、设备台账查询、历史故障案例问答 | 低 — 与一般企业知识库相当 |

**原方案没显式分这三层,把 5 条 MANDATORY 笼统套给所有 skill,这本身就是漏洞** —— 对 C 类 skill 强行要求"机理硬约束"或"闭锁绑定"是过度工程,对 A 类 skill 仅要求"可解释输出"是远远不够。下文每条会标注其真正适用层。

---

## §1. 既有 5 条逐条验证

### 1.1 机理 + AI 双引擎硬约束

**判定**: ✅ **MANDATORY (A 类、B 类必需;C 类可豁免)**

**理由**:
电力系统是物理系统,有不可违反的硬约束:基尔霍夫定律、热稳极限、暂态稳定极限、N-1 准则。LLM 是统计模型,在 OOD(分布外)场景下完全可能给出"违反基尔霍夫"的方案而置信度看起来还很高。如果 AI 输出能直接覆盖机理结果,这是**根本性架构错误**。

**反例 / 业务事故**:
- 大负荷低电压情境下,LLM 基于历史相似日推荐"投入电容器组 C1+C2",但 C2 当时正在检修。机理引擎(电网拓扑实时模型)知道 C2 不可用,如果 AI 输出能压机理输出,操作票就会下发到一台冷备设备,现场拒动 → 调度员被迫人工切负荷 → 上经济损失级事故。
- 历史类比:2003 美加大停电、2012 印度大停电 ——根因都是局部决策违反系统级约束。AI 没有理由比人类调度员更"敢"。

**精炼建议**:
- "硬约束不可被 AI 覆盖" 应写成可执行的:**机理引擎 verdict 与 AI 输出冲突时,机理 verdict 单调胜出,且冲突事件 MUST 入审计日志**。原方案只说"硬约束",没说"谁裁决冲突"。
- C 类 skill(纯查询)豁免本条,因为它们不输出控制建议。

---

### 1.2 三重校核(pre/process/post)

**判定**: ✅ **MANDATORY (A 类、B 类必需)**,但**原方案的"三重"切分是冗余的工程包装**

**理由**:
拦截非法/危险输入和输出本身是必要的。但"pre/process/post"这种"三重"叙事是 AI 文风,真正的本质是**两道不可省的关口**:
1. **输入侧合规校验**(权限 + 数据质量 + 任务边界)
2. **输出侧物理校验**(N-1 / 潮流 / 限额)

把"process_check"单列为第三重在工程上意义模糊 —— 对 LLM 来说"过程"是黑盒,真正可校的是输入和输出。

**反例 / 业务事故**:
- 没有输出侧校核:AI 推荐"500 kV 某线路开断",直接生成操作票。但该线路开断后另一回线潮流 105% 长期稳定极限,N-1 失效 → 后续任何故障都可能发展成系统瓦解。
- 没有输入侧校核:某未授权 skill 通过 prompt injection 调用了控制类 skill → 越权操作。

**精炼建议**:
- 写成 **"输入校核 + 输出校核两道闸,缺一不可"**,而不是"三重"。
- 输出校核 MUST 调用机理引擎(潮流计算、N-1 扫描),**不可由 LLM 自评**(自评 = 让裁判员同时当运动员)。

---

### 1.3 强可解释输出

**判定**: 🟡 **SHOULD,不是 MANDATORY** ——除非配套的法定追责机制存在

**理由**:
"引规程条款 + 量化校核 + 影响范围"听起来很对,但单纯写在 skill 标准里**没法强制执行**。LLM 可以编造一个"看起来像规程条款"的引用(GB/T 31464-2015 §6.3 —— 这条号可能根本不存在),如果没有下游做引用真实性校验,这一条等于装饰。

更核心的是:**可解释性是结果,不是输入**。真正应该 MANDATORY 的是 **"决策溯源链"**:每个调控建议必须能回答"输入数据是什么 + 调用了哪些工具 + 机理校核给了什么结果 + LLM 生成时模型版本/温度/seed 是什么"。能回答这些,自然能输出解释;不能回答,再花哨的解释都是事后编造。

**反例 / 业务事故**:
- 假可解释:LLM 输出"根据 GB/T 31464-2015 §3.2.1 应执行此操作",但该条款不存在 —— 无校验机制,事故复盘时审计组找不到依据 → 责任认定卡死。
- 真问题:发生事故,调度员主张"AI 让我做的",AI 提供商主张"调度员有最终决定权",**没有溯源链就无法分责**。

**精炼建议**:
- 把这条**降级为 SHOULD,然后 MANDATORY 一条新的"决策溯源链"**(见 §2 NEW-M3)。
- 可解释字段(规程引用、量化结果)作为 SHOULD,且引用条款 MUST 来自受控规程库(下游真实性校验,见 NEW-M9)。

---

### 1.4 闭锁绑定

**判定**: ✅ **MANDATORY (仅 A 类控制类 skill)**

**理由**:
这条不仅是合规要求,也是**电力调度行业现行规程已经强制的**(操作票 + 双人复核 + 闭锁逻辑是 N 多年的运行实践)。AI skill 进入控制环不能架空既有闭锁机制。

**反例 / 业务事故**:
- AI skill 输出"开断 500 kV 某线",绕过常规挂牌闭锁直接下发 → 现场带电检修人员未撤离 → **人身事故**。这是行业最敏感的红线。
- 没有闭锁绑定的 AI 控制 skill 在监管上是无法投运的 —— 国家电网调度控制管理规程明确要求所有调控指令受闭锁逻辑约束。

**精炼建议**:
- 必须**显式说明"AI skill 不能创建新的闭锁条目,只能消费既有闭锁系统的判决"** —— 这是常被忽略的边界。AI 作为"新增智能体"绝不能给自己开闭锁后门。
- 这条措辞要从"绑定"升级到 **"闭锁系统作为外部权威,AI skill 仅作 client,不可作 issuer"**。

---

### 1.5 时效分级

**判定**: 🟡 **降级到 SHOULD** —— 原方案这条说得不到位

**理由**:
"秒级 AGC / 分钟级潮流 / 小时级日前 / 日级检修"是对的,但更深的问题原方案没说:**LLM 在闭环秒级控制中根本不能用** —— 不是分级问题,是**根本不适合**的问题。一次 LLM 调用最快 200ms,带网络抖动到几秒,而 AGC 周期通常 4-8 秒,任何一次 LLM 抖动都会导致控制环断裂或震荡。

时效分级真正需要 MANDATORY 的不是"分四档",而是 **"AI skill 不得部署在闭环秒级控制路径上"**(明确禁区)。

**反例 / 业务事故**:
- 错把 LLM-driven skill 嵌入 AGC 闭环,LLM 调用偶发 timeout → AGC 控制器收不到 setpoint 更新 → 一次频率扰动期间响应缺位 → 联络线功率越限 → 触发解列。
- 这就是为什么实际应该用 **强化学习 + 硬阈值兜底** 做秒级控制,LLM 留在分钟以上的"参谋"层。

**精炼建议**:
- 改写成:**"延迟硬上限 + 秒级闭环禁区"**(见 §2 NEW-M5)。
- 时效"分级"作为编排提示降为 SHOULD。

## §2. 漏点扫描 — 应该新增的 MANDATORY

下面 10 条是原方案完全没覆盖、但在电力调度场景一旦缺失会出生产事故/监管不合规的要素。每条独立判断 MANDATORY 强度。

### NEW-M1: 量测数据质量与时间戳一致性

**判定**: ✅ **MANDATORY (A 类、B 类)**

**理由**:
所有调度决策的输入是 SCADA 量测数据,但量测数据**天然带不可靠性**:数据 stale(传感器掉线)、数据 bad(校验位失败)、时间戳跳变(主备切换 jitter)、PMU 不同步。AI skill 必须能识别这些"输入不可信"状态并拒绝决策,而不是当成正常数据处理。

**反例 / 业务事故**:
- 某变电站 RTU 通信中断 30 分钟,SCADA 数据冻结但状态位未刷新。AI skill 看见"潮流稳定"就给出"维持现方式"建议。实际线路在断面已超载 5%。无 stale 检测 → 错失干预窗口。
- 主备切换瞬间 SCADA 时间戳出现 2 秒回跳,AI 把"5 分钟前的功率值"当作"现在的"做潮流外推 → 决策依据是 5 分钟前的电网,不是现在的电网。

**为什么原方案漏了**:
原方案站在"AI 输出"角度想问题,没站在"AI 输入"角度想。但 garbage in, garbage out,输入不可信比输出不准更危险 —— 输出不准还能被输出校核拦下,输入不可信连校核都校不出来(因为校核器的输入也是同一份脏数据)。

**MUST 化措辞**:
A/B 类 skill MUST 声明所依赖的量测点列表,平台在每次调用前 MUST 注入数据质量元数据(timestamp / quality_flag / staleness_seconds);任一关键量测 stale 超阈值或 quality bad,skill MUST 返回 `INPUT_QUALITY_FAILED` 错误,不得继续推理。

---

### NEW-M2: 决策法定签名与不可抵赖审计

**判定**: ✅ **MANDATORY (A 类必需,B/C 类 SHOULD)**

**理由**:
电力调控指令在监管层面是**有法律效力的命令**,谁下发、谁复核、谁执行有法定追责链条。AI skill 进入这一链条必须给出**可法庭采信**的签名 —— 不是 LLM 自带的"我建议",而是带数字证书 + 时间戳 + 哈希链的不可抵赖记录。

**反例 / 业务事故**:
- 发生设备损毁事故,调查组要求复盘"是谁批准了这次操作"。AI skill 日志只有 "agent_id=xxx, prompt=yyy, reply=zzz" → 法律上无法认定 AI 提供商责任,也无法证明操作员"未充分复核",最终被迫按"无法定责"结案,留下监管空白。
- 这一条是**电力监管入网评审必查项** ——没有不可抵赖审计的 AI skill 根本拿不到投运批准。

**为什么原方案漏了**:
原方案"权限分级"只说了访问控制,没说**事后追责**。这两个不是一回事 —— 访问控制管"谁可以做",签名审计管"做了之后是谁做的、改不了"。

**MUST 化措辞**:
A 类 skill 每次输出 MUST 携带 `(agent_signature, model_version, prompt_hash, evidence_chain_hash, utc_timestamp)` 五元组,签名 MUST 使用电力调度专用 PKI(国密 SM2)证书;日志 MUST 进入 WORM 存储,保留期 ≥ 国家电网调度日志法定保留年限(常见 10 年以上)。

---

### NEW-M3: 决策溯源链(完整证据 chain)

**判定**: ✅ **MANDATORY (所有类别)**

**理由**:
比可解释性更基础的是**溯源**。LLM 是黑盒但 skill 不应该是黑盒。任何一次 AI 输出必须能逐步回放:输入数据快照 → 调用工具列表(及其入参/出参)→ 机理引擎 verdict → LLM 生成参数 → 最终输出。**断了链条就等于断了责任**。

**反例 / 业务事故**:
- 一次 AI 推荐造成调度员误操作。事故复盘时,要重现 AI 当时"看到了什么数据、做了什么校核",但日志只存了 prompt + reply,RAG 检索结果丢失,机理校核结果丢失 → 无法判断是 AI 错、数据错还是校核器错 → 责任认定僵局,监管处罚转嫁到调度员个人。
- 这是 §1.3 真正想要表达的核心,但原方案把"可解释"和"可溯源"混在一起。可解释是**面向人**的输出文案;溯源是**面向监管+事故复盘**的内部 chain,后者不可省。

**为什么原方案漏了**:
原方案"强可解释输出"试图覆盖,但实际上**只盯了输出文案,忽略了内部 evidence chain**。

**MUST 化措辞**:
每次 skill 调用 MUST 生成 evidence chain 存档,包含:input_snapshot_id(SCADA 量测 hash 引用)、tool_call_sequence(顺序+入参+返回)、mechanism_check_results(机理引擎结论)、llm_generation_params(model/temperature/seed)。chain 内任意环节缺失 → skill 调用 MUST 标记为 `evidence_incomplete` 不得入正式记录。

---

### NEW-M4: 通信退化下的本地降级与黑启动行为

**判定**: ✅ **MANDATORY (A 类必需)**

**理由**:
电力系统会发生"通信退化"乃至"全系统瘫痪需黑启动"等极端工况。在这种工况下:
- 主站到 LLM 服务的网络可能断
- 云端 LLM 提供商可能不可达
- 调度数据网可能降级到带宽极小

如果 AI skill 在这些场景下**没有明确降级行为**,会出现"AI 一沉默,调度员习惯性等 AI 建议而错失干预窗口"或"AI 异步回包延迟到秒级,把决策路径锁死"等问题。

**反例 / 业务事故**:
- 一次大停电发生,调度数据网带宽降到 1%,云端 LLM 调用超时。AI skill 既不返回 fallback 也不明确 fail-fast,调度员等 30 秒才发现 AI 不可用,错失了关键的孤网频率控制窗口。
- 黑启动场景下电网拓扑不稳定,AI 训练时从未见过这种 "电网半瘫痪" 状态,推理结果完全不可信 —— 这种场景下 AI 不应该"努力给答案",应该**显式投降**:"当前工况超出我的可信范围,请人工接管"。

**为什么原方案漏了**:
原方案默认 AI skill 在线就能用,没考虑通信退化和黑启动这种"全行业高敏场景"。这是电力行业相对其他行业最特殊的需求之一(几乎没有别的行业要求"主系统瘫痪时智能体如何降级")。

**MUST 化措辞**:
A 类 skill MUST 声明 `degradation_policy`:在网络/上游服务超过阈值秒数不可达时的行为(fail-fast / local-cached-fallback / explicit-handoff-to-operator),且 MUST 显式声明"超出可信工况边界"的识别条件(如电网拓扑突变、断面失稳)及对应的"投降"输出。

---

### NEW-M5: LLM 延迟硬上限 + 闭环禁区

**判定**: ✅ **MANDATORY (秒级控制路径)**

**理由**:
原方案 §1.5 时效分级把这一条说成了"分级",但真正的硬约束是**禁区**:LLM 不能进入秒级闭环。这不是"应该尽量快",而是**架构禁区**。一次 LLM 调用 P99 延迟在秒级是常态,而 AGC、AVC、紧急控制、稳控装置的控制环周期都在毫秒到秒级 —— 物理上不兼容。

**反例 / 业务事故**:
- LLM-driven skill 错部署在 AGC 二次调频闭环里,某一次模型推理超时 5 秒,导致 4 个 AGC 周期没有 setpoint 更新,联络线功率越限触发解列保护 → 区域电网解列。

**为什么原方案漏了**:
原方案"分四档"把秒级和分钟级并列叙述,没说"秒级是禁区"。这会误导厂商把 LLM 硬塞到 AGC 链路。

**MUST 化措辞**:
LLM-backed skill MUST 声明 `latency_class`,其中:
- `closed_loop_seconds`(秒级闭环):**禁止使用 LLM 推理**,仅允许用规则引擎/RL 策略
- `advisory_minutes`(分钟级建议):允许 LLM,P99 延迟 ≤ 10 秒
- `planning_hours_or_more`(小时级以上):允许 LLM,无延迟硬约束

平台 MUST 在调度入口校核 skill 声明的 latency_class 与挂载位置匹配,违反 → 调用拒绝。

---

### NEW-M6: 跨调度层级权限与数据边界

**判定**: ✅ **MANDATORY (国调/网调/省调/地调/县调五级架构必需)**

**理由**:
中国电力调度分国调/网调/省调/地调/县调五级,**每一级对电网的调管范围严格隔离**:省调不能直接遥控县调辖区设备,反之亦然。AI skill 跨层级时如果不带边界,极容易越权(尤其是 LLM 上下文中混入了多层级数据时,容易"越想越大",生成跨层级指令)。

**反例 / 业务事故**:
- 省调一个负荷预测 skill,在生成"明日运行方式"时引用了某个 220kV 站(地调辖区)的检修计划,结果建议"开断 110kV 某线"——这条 110kV 线属于地调辖区,省调无权指挥。指令下发现场会被拒,但更严重的是**这个建议本身泄露了越权倾向**,在监管检查时是重大不合规。
- 反向:一个跨省互联场景下,A 省调 skill 调用了 B 省调的实时数据(应当通过国调中转),违反《电力监控系统安全防护规定》数据流向控制。

**为什么原方案漏了**:
"权限分级"只是粗粒度的"角色权限",不等于细粒度的"调度层级 + 调管范围"。后者是电力行业独有的、高度结构化的边界,标准里必须显式列。

**MUST 化措辞**:
A/B 类 skill MUST 声明 `dispatch_level: {national | regional | provincial | municipal | county}` + `controllable_assets_filter`(基于电网拓扑模型的可控资产列表)。任何输出涉及非声明范围资产 → MUST 标 `out_of_scope` 拒绝。跨层级数据访问 MUST 走调度数据网纵向加密通道,不得旁路。

---

### NEW-M7: 模型版本锁定 + 行为漂移管控

**判定**: ✅ **MANDATORY**

**理由**:
LLM 提供商升级模型是常态,但**对于调度场景,任何一次模型升级都等于换了一次"AI 调度员"**。新模型可能推理倾向不同、温度参数响应不同、对相同提示给出不同结果。如果不锁版本,昨天测试通过的 skill 今天可能就突然行为漂移。

更隐蔽的是:**OpenRouter / API 提供商可能在不通知用户的情况下后端切换模型(我们项目自己的 MEMORY 里就记录过 OpenRouter 路由动态性导致 capability 漂移问题)**。调度场景对这种"沉默漂移"零容忍。

**反例 / 业务事故**:
- 某 skill 在 model-X-2025-Q1 上验证通过投运。半年后 provider 静默升级到 model-X-2025-Q3,行为微调。某次工况下输出更激进 → 人工复核没发现 → 出现操作偏差。事后查日志,发现"是同一个 skill 但模型版本悄悄变了",这种事故责任认定极难。

**为什么原方案漏了**:
原方案是 Skill 静态结构标准,完全没考虑"AI 后端在变"。但这是 LLM 时代和传统调度软件最大的差别 —— 传统软件版本是离散的、显式发布;LLM 后端可以滚动升级、对调用方透明。

**MUST 化措辞**:
A/B 类 skill MUST 在元数据声明 `model_lock: {provider, model_id, version, fingerprint}`;每次调用 MUST 校验实际响应中的模型 fingerprint 与声明一致,不一致 → 拒绝输出 + 告警。模型升级 MUST 走"重新评测 + 回放历史 case + 调度专家复审"的流程,不能默认继承上一版本投运批准。

---

### NEW-M8: 投运前历史 case 回放与极端工况扫描

**判定**: ✅ **MANDATORY (A 类、B 类必需)**

**理由**:
任何 AI skill 进入调度生产环境之前,必须证明它在**电网历史发生过的极端工况**下不会出错。这不是"测试覆盖率"那种工程概念,而是行业特定的:**这条线路 N-1 这条断面失稳过、那个变电站全停电过、这个新能源场站脱网过** —— skill 必须在这些 case 上回放并通过。

**反例 / 业务事故**:
- 某新建 skill 在常规工况下表现优秀,但从未在"夏季用电高峰 + 极热天气 + 新能源大幅波动"复合工况下测试过。第一次遇到这种工况,skill 给出明显失误的建议。
- 历史教训:8.14 美加大停电的根本原因之一就是"系统在某种极端组合工况下从未被测试过"。AI 不能让这个教训重演。

**为什么原方案漏了**:
原方案有"测试用例"作为通用要素,但**没有"必须回放电网历史极端 case"**这条特殊性。一般行业的测试覆盖率不能替代调度行业的"已知极端工况扫描"。

**MUST 化措辞**:
A/B 类 skill 投运 MUST 通过国调中心维护的 **历史极端工况库回放测试**(具体覆盖项由标委会维护:典型大停电 case、典型 N-2 失稳 case、典型新能源大幅扰动 case 等)。回放结果 MUST 作为投运批准前置材料归档。新增极端 case 入库后,既有 skill MUST 在合理周期内重新通过回放。

---

### NEW-M9: 规程引用与知识库受控更新

**判定**: ✅ **MANDATORY (引用了规程的 skill 必需)**

**理由**:
电力调度规程(国家电网公司调度控制管理规程、安稳导则、两个细则等)是**有版本、有发布日期、有修订记录**的法定文档。AI skill 引用规程时:
1. 引用必须**对得上当前生效版本**(已废止条款不能引)
2. **规程修订时,引用了该条款的所有 skill 必须能被反查**(知道哪些 skill 受影响)
3. RAG 知识库**不能让任意人写入**,必须有受控更新机制

**反例 / 业务事故**:
- 某 skill 引用了 GB/T XXXX-2018 §6.3,但该条款在 2024 年修订版已废止/调整。skill 输出时仍按旧条款 → 事故复盘时发现"规程已变 18 个月,AI 没跟上"。
- RAG 知识库被误更新,塞入了一份未经审核的内部材料,LLM 检索时优先引用该材料,导致输出引证错误。

**为什么原方案漏了**:
原方案"强可解释输出"提到引规程,但**没说规程库本身怎么管**。这是看似细节但极其关键的合规漏洞。

**MUST 化措辞**:
A/B 类 skill 引用规程 MUST 来自国调中心受控知识库(由标委会授权方维护),引用 MUST 携带 `(regulation_id, version, clause_id, library_snapshot_hash)`;规程修订 MUST 触发反查所有引用 skill 的影响评估;RAG 知识库写入 MUST 经规程编辑工作流(双人审批 + 版本记录)。

---

### NEW-M10: 多 skill 编排时的优先级与互斥

**判定**: 🟡 **SHOULD,在多 skill 大量并发部署后升级到 MANDATORY**

**理由**:
当多个 AI skill 并行运行(负荷预测 + 检修计划 + AGC 辅助 + 故障诊断 ...),它们对同一资源(同一断面、同一设备组)可能产生**冲突建议**。如果没有明确的优先级和互斥规则:
- 哪个 skill 的建议先采纳?
- 两个 skill 同时建议对同一断面操作时谁让谁?
- 紧急控制 skill 和检修计划 skill 冲突时如何抢占?

**反例 / 业务事故**:
- 检修计划 skill 输出"明日 9 点 110kV 某线停役";同时,故障诊断 skill 在 8:55 输出"立即开断同一线路隔离故障"。两个建议都达到调度员屏幕,优先级不明 → 误操作风险。

**为什么没列 MANDATORY**:
当前调度 AI skill 部署还在初期(单点应用为主),实际并发冲突场景少。但当国调中心规模化部署后(预期 1-2 年),这条会从 SHOULD 升级到 MANDATORY。**当下作为 SHOULD 显式列出,提醒厂商提前设计**。

**MUST/SHOULD 措辞**:
SHOULD:多 skill 协同部署的平台 SHOULD 维护 skill 优先级与互斥矩阵,冲突建议 MUST 升级到调度员人工裁决。

---

## §3. SHOULD 级候选(不到 MANDATORY,但应该写入规范)

不重复 §1.3、§1.5、§2 NEW-M10 中已降级到 SHOULD 的项,补列以下 5 条。

| ID | 名称 | 一句话理由 |
|----|------|-----------|
| S1 | LLM 输出温度/采样参数固化 | 同一 skill 不同 temperature 行为差异巨大,SHOULD 锁参数 |
| S2 | Confidence calibration | skill 输出 SHOULD 附"我的置信度",且置信度低于阈值时显式标"低信心建议",不能让所有输出看起来都同样自信 |
| S3 | 反例/honeypot 测试用例覆盖 | skill 投运测试 SHOULD 包含恶意输入/对抗 prompt 注入 case,但很多场景下做不到完整覆盖,降为 SHOULD |
| S4 | 跨 skill 数据隔离 | 检修计划 skill 不应直接读到实时控制 skill 上下文,SHOULD 在 platform 层做隔离;并发部署初期可松 |
| S5 | A/B 测试与 canary 发布 | 新 skill SHOULD 先小范围试点(如某地调先用),再扩大;但具体灰度比例由各局自定,不强制 |

---

## §4. 最终建议的 MANDATORY 清单(合并 §1 + §2)

按"出事故/不合规的硬度"排序,**前 7 条是真正的电力调度 skill MANDATORY**:

| 序 | 名称 | 一句话定义 | 适用类别 |
|----|------|-----------|---------|
| **M1** | **机理引擎单调胜出** | 机理与 AI 冲突时机理 verdict 单调胜出,且冲突 MUST 入审计 | A/B |
| **M2** | **输入侧 + 输出侧两道校核闸** | 输入做权限+数据质量;输出做物理校核(N-1 / 潮流 / 限额),不允许 LLM 自评 | A/B |
| **M3** | **闭锁系统作为外部权威** | 控制类 skill 仅作闭锁系统 client,不可作 issuer;无 token 拒绝 | A only |
| **M4** | **数据质量与时间戳一致性** | 量测数据 stale/bad 时 skill MUST 拒绝推理,而非"硬着头皮算" | A/B |
| **M5** | **决策溯源链(evidence chain)** | input_snapshot + tool_calls + mechanism_check + llm_params 全链留痕,缺一项→标 incomplete | All |
| **M6** | **法定签名 + WORM 审计** | 每次输出带国密签名+时间戳+哈希,WORM 存储 ≥ 法定保留年限 | A only |
| **M7** | **跨调度层级权限边界** | skill 声明 dispatch_level + controllable_assets_filter,越界拒绝 | A/B |

**接下来的 3 条评级为"高优先级 MANDATORY,但可分阶段强制"**(给行业一个落地缓冲期):

| 序 | 名称 | 备注 |
|----|------|------|
| M8 | LLM 延迟硬上限 + 秒级闭环禁区 | 现阶段先以"禁止部署在 AGC/紧急控制路径"宣示;延迟 SLA 细则后续完善 |
| M9 | 模型版本锁定 + 漂移管控 | 现阶段先要求 fingerprint 校验;升级评测流程后续标委会发指引 |
| M10 | 投运前极端工况回放 | 历史 case 库由国调中心牵头建,现阶段以"声明已回放"+ 抽查为主 |

**为什么不是 5 条 / 不是 12 条**:
- 5 条:覆盖不全,把"数据质量"、"溯源"、"签名审计"漏掉是把整个监管链放在裸奔状态。
- 12 条以上:超过人能记得住的硬约束阈值,会被当作"建议"而非"红线",落地时第一批被砍。
- 7 条核心 + 3 条分阶段是合理的折中:**前 7 条是"不满足就拿不到投运批准";后 3 条是"投运后逐步完善"**。

**原方案 5 条做了什么调整**:
- 1.1 机理双引擎 → 保留 + 措辞强化为 M1
- 1.2 三重校核 → 简化为 M2(两道闸)
- 1.3 强可解释 → 拆为 M5(溯源,MANDATORY)+ S2(置信度,SHOULD);可解释文案降到 SHOULD
- 1.4 闭锁绑定 → 保留 + 措辞强化为 M3(强调 client/issuer 边界)
- 1.5 时效分级 → 拆为 M8(秒级禁区,分阶段 MANDATORY)+ 时效"分级"作为编排提示降为 SHOULD

**新增的 5 条 MANDATORY**:M4 数据质量、M5 溯源、M6 签名审计、M7 跨层级边界、M9 模型锁定、M10 极端 case 回放。

## §5. Open Issues — 留给后续讨论

1. **不同调度层级 MANDATORY 是否相同?**
   - 国调/网调:M1-M10 全套
   - 省调/地调:M1-M7 必,M8-M10 推荐
   - 县调:由于资源限制,可能只能强制 M1/M2/M3/M4 + 引用上级 skill;独立部署 LLM 资源不现实
   - 这一裁剪需要分层级专家共识,不是技术问题

2. **"机理引擎单调胜出"在新能源场景边界**
   - 新能源大规模接入后,部分机理模型(传统稳定计算)本身已不够准 —— 此时 AI 数据驱动模型可能比机理更接近真值。这种场景下"机理单调胜出"反而成为劣势。
   - 需要电力专家明确:**哪些场景仍是"机理硬约束",哪些场景需要"机理 + AI 投票/加权"**。本研究维持"机理单调胜出"作为默认值,但应允许标委会评审通过的特定场景例外。

3. **国密 SM2 签名是不是过度?**
   - 强制 SM2 会把所有非国产 LLM 服务挡在门外(对自主可控好,对生态接入坏)
   - 替代方案:允许多种符合电监会要求的签名方案(SM2 + RSA-PKCS1 + ECDSA),由各局选择
   - 这是政策决策,不是技术决策

4. **WORM 存储法定保留年限**
   - 调度日志保留要求各文件不一致,公开能查到的常见是 5-10 年。具体年限需电力专家+法务联合给定,不能拍脑袋。
   - 本研究列"≥ 法定保留年限"作占位

5. **极端 case 库谁来维护**
   - 国调中心牵头是合理的,但要避免"case 库变内部黑名单,新厂商无法进入"
   - 建议 case 库脱敏后向认证厂商开放,并设申诉通道

6. **本研究多条判断未独立 verify**
   - 未联网核对国标条款号(GB/T 31464、DL/T 510、GB 38755 等具体条文细节)
   - 未联网核对调度数据网纵向加密 + 电力监控系统安全防护具体条款
   - 未核对国家电网调度日志 WORM 法定保留年限的最新精确值
   - 这些**正式发布前必须由电力专家+法务复核**,本研究只做业务判断

7. **多 skill 编排冲突优先级**(NEW-M10)
   - 当前列 SHOULD,但这一条的 MANDATORY 化时机比较微妙 —— 太早会过度工程,太晚会发生事故才补
   - 建议在国调首批 5+ skill 并发部署 6 个月后做规模评估
