## B. User 先验定位(2026-04-26 直接陈述,非 audit)

### B.1 产品 / 受众定位

- **EAASP 是面向企业的**(B2B 平台层)
- **Grid 是通用的**(开发者 / 单租户 / 桌面 / 工具向)
- 二者**受众不同**,不是同一产品的两种渠道

### B.2 工时倾向

- **两个都想做**,不二选一(排除 Phase 4 简单选项)
- **重心在 Grid** —— user 自己工时主要投 Grid
- **EAASP 内部职责切分**(2026-04-26 socratic 精准化):

  | 层 | 你专心做 | 他人接手 |
  |----|--------|--------|
  | **L0 Protocol** —— `proto/eaasp/runtime/v2/` | ✅ | — |
  | **L1 Runtime** —— Grid (`grid-engine`/`grid-runtime`) | ✅ 主战场 | — |
  | **L2 内存与技能** —— `tools/eaasp-l2-memory-engine` + `eaasp-skill-registry` + `eaasp-mcp-orchestrator` 的**基础引擎** | ✅ 基础组件 | 数据(客户语料 / vector 库 / 企业知识) |
  | **L3 治理层** —— `tools/eaasp-l3-governance` 的**policy DSL + risk classification 引擎** | ✅ 基础组件 | 企业 policy 数据 / 集成第三方治理系统 |
  | **L4 编排层** —— `tools/eaasp-l4-orchestration` 的**session lifecycle / SSE / orchestration 引擎** | ✅ 基础组件 | 与企业内部系统集成(SSO / 工作流 / 通知 等) |
  | **数据层 + 集成层**(横切) —— 客户数据接入 / 企业内部系统对接 / SaaS 平台集成 / SSO / webhook / 第三方 API 适配 | — | ✅ 他人主要做这块 |

  **结构化解读**(2026-04-26 socratic 精准化):
  - **你专心做**: 各层的**引擎** + **Grid 全栈** —— 即 L0 protocol、L1 runtime、L2/L3/L4 各自的核心 engine
  - **他人主要做**: 跨层的**数据 + 集成**横切关注 —— 客户数据怎么进、企业系统怎么对接、第三方集成、SSO 等
  - 切分逻辑: **"engine"(可复用核心) vs "data/integration"(场景特定)**,不是按层切,也不是按技术/商务切

### B.3 二选一倾向(若被迫)

- 二选一的话**选 Grid**
- 目标:**Grid 能很好支撑企业级 AI 应用**(隐含约束 —— Grid 不能为了通用化而舍弃企业级品质)

## C. 对 ADR-V2-023 框架的 baseline 偏离

### C.1 Leg 与"engine vs data/integration"的语义错配

- ADR-V2-023 原模型用 **Leg A(EAASP 集成)/ Leg B(Grid 独立)**切产品形态
- User 实际心智模型用 **"engine vs data/integration"**切职责:
  - **engine 层(全栈基础组件): 你专心做** —— L0 Protocol + L1 Grid + L2 memory/skill/MCP + L3 governance + L4 orchestration 的引擎部分
  - **data + integration 层: 他人主要做** —— 客户数据接入 / 企业内部系统对接 / SSO / 第三方 API 适配 / 跨层场景化集成
  - **Grid 全栈 + EAASP 基础组件**都是你的工时主战场,不是 "Grid primary, EAASP dormant"
- 这与 ADR-V2-023 字面 "Leg A active / Leg B dormant" 措辞**正交**(不是反向,是切角不同 —— 一个按"产品形态"切,一个按"engine vs data/integration"切)

### C.2 §P5 触发条件需要重新解读

- §P5 4 条 Leg B 激活触发条件原本从"对冲外部 EAASP 风险"角度写,假设 EAASP 是外部 / 上游
- 如果 EAASP 是同仓孵化(基础组件由 user 自己做),§P5 措辞需要改:
  - 第 1 条 "EAASP 项目延期 / 停滞 / 定价不合理" 变成**自评**(自己控制的 EAASP 节奏),语义弱化
  - 第 2 条 "≥2 客户拒绝走 EAASP 直接买 Grid" 仍有效,但前提是 Grid 已商品化到能直接卖
  - 第 4 条 "团队富余 ≥30%" 在 user 心智中**默认成立**(Grid 一直就是工时重心),不需要等
- **行动**: Phase 4.1 audit 必须重新框定 §P5,可能产出 ADR-V2-024 调整 ADR-V2-023 框架(以及 §P5 触发条件 / Leg 命名)

### C.3 Phase 4 决策的真实形态

不是 "走 Leg A 还是 Leg B 还是双腿",而是更结构化的:

1. **engine vs data/integration 切分是确认事实**(B.2 表已锁),audit 不需要再决基础切分
2. **EAASP 同仓孵化的 graduation criteria**:
   - 何时分仓?分仓的完成度门槛是什么?
   - 分仓后"data/integration 层接手者"对接面: contract-v1.1 是否够,还是需要新的 extension / SLA / 数据格式契约?
3. **Grid 自身产品化路径**:
   - `grid-cli` / `grid-server` / `grid-desktop` / `grid-platform` 哪些先发力?
   - 通用化 vs 企业级品质如何权衡(B.3 隐含约束)?
4. **engine 与 data/integration 的边界协议**:
   - "他人主要做 data/integration" 不等于 "我们与他们之间无契约"
   - 需要定义:engine 暴露哪些 hook / extension point / config 表面 / 数据接口 给 data/integration 层用?
   - 这与 contract-v1.1(L1 over gRPC)是不同抽象层,L2/L3/L4 各自可能需要自己的 data ingest 契约 / integration extension contract

## D. Sanity check guards for Phase 4.1 audit

audit 阶段如果给出以下结论,必须显式说明为什么 baseline 错了:

- ❌ "Grid 应当 dormant,集中精力做 EAASP" → baseline 强烈相反(Grid 是工时主战场),**audit 必须给硬证据**
- ❌ "EAASP 不该分仓,永远同仓" → baseline 已说 "未来分仓",**audit 必须解释为什么放弃这个计划**
- ❌ "Grid 不再支撑企业级" → baseline 隐含 B.3 约束,**audit 必须说明放弃企业级的代价**
- ❌ "EAASP 引擎层(L2/L3/L4 基础组件)交给他人做" → baseline 明确 user 自己做这部分,**audit 必须给硬证据为什么改变切分**
- ❌ "data/integration 层 user 自己也做" → baseline 已明确他人主要做这块,**audit 必须解释 user 工时为何转向这里**

audit 给出以下结论是 baseline 一致(不需要硬证据):

- ✅ Grid 全栈 + EAASP 引擎层(L0-L4 基础组件)由 user 专心做
- ✅ data + integration 层(客户数据 / 企业系统对接 / SSO / 第三方 API 等)由他人主要做
- ✅ EAASP 同仓孵化 → 未来分仓
- ✅ Grid 产品化优先级在 cli / server / desktop / platform 中分轻重
- ✅ engine 与 data/integration 之间的边界契约(extension point / 数据格式)需要 audit 进一步定义

## F. 调度 skill 规范 v0.1 评估浮现的 audit agenda(2026-04-26 追加)

> **来源**: `docs/external-review/2026-04-26-eaasp-skill-spec-coverage-internal.md`(commit c6dadbc)— 把电力调度 AI 智能体 Skill 规范 v0.1 的 MANDATORY 全集对照 EAASP 7 个组件,识别出 4 / 7 / 13 / 6 项的 🟢/🟡/🔴/⚫ 分布
>
> **§F 与 §E 的关系**: §E 来自 socratic 抽象问题,§F 来自规范评估具体清单。§E 5 条问题是"audit 要回答的元问题",§F 4 条 Q 是"§E §C.1 'engine vs data/integration' 切分被规范评估具体化后浮出的真问题"。**不是平行清单,§F 是 §E §C.1 的具体化镜像**。
>
> **重要边界**: §F 是 audit agenda(问题),不是 backlog(动作)。回答这 4 个 Q **不**预设要做哪些事,只是 audit 必须显式给出立场,否则 13 项 🔴 / 7 项 🟡 永远卡在"该谁做"的悬置态。

### F.1 audit 必须显式回答的 4 个前置问题

#### Q1: EAASP 要不要做调度行业 vertical?

规范 13 项 🔴 里**有 ~8 项**(M1 机理 / M3 闭锁 / M7 dispatch_level / M8 latency_class 字段 / M9 灵活字段 / 6.6 多 skill 优先级 / Gotchas 段语义 / 风险等级字段)的归属取决于这个回答:

- **答"是"**: EAASP engine 直接吸收这 8 项,引入 dispatch_level / risk_class / latency_class 字段 + 引擎语义("verdict 来源优先级" 作为 L3 policy DSL 一类规则)。**代价**: EAASP 不再行业中立,未来要做金融 / 医疗 vertical 时同样的扩展点要再 abstract 一次。
- **答"否"**: EAASP 提供**扩展点**(schema 命名空间 / hook 接入点 / event 类型),调度 vertical 由厂商在 EAASP 之上写。**代价**: EAASP 工程量小但要精心设计扩展点。
- **答"两者并行"**: 哪些字段 / hook MUST 在 engine,哪些在 vertical?**这是真正难的事**,要逐项判断。

**与 §C.1 的关系**: 这是 §C.1 "engine vs data/integration" 切分**在调度行业场景下的具体化** —— "vertical 层"是 §C.1 "他人接手数据/集成"的一个具体形式(只不过这个"他人"可能是厂商,不是泛指数据接入方)。

**与 §E.5 的关系**: 答"是" / "并行" 大概率触发 ADR-V2-024 修订(语义反转加强);答"否" 弱化 §P5 解读紧迫度。

#### Q2: 13 项 🔴 里另外 ~5 项(governance workflow)是否要做?

R1-R9 lint / HITL workflow / lifecycle 5 阶段 / 反措 → skill 升级流水 / 多维分类索引 — 这些**可以不绑定调度行业**,是任何企业 agent 平台都需要的:

- **答"做"**: EAASP 从"通用 agent 平台基础引擎"升级为"企业级 agent governance 平台"。**这是产品定位升级**,会拉高研发成本 + 拉长 time-to-market。
- **答"不做"**: EAASP 留在"基础引擎"层,governance workflow 由上层应用 / vertical 自己解决。**风险**: 没有 governance 的 agent 平台在企业市场卖不动(规范 §8 论述)。

**与 ADR-V2-023 Leg A vs Leg B 的关系**: governance workflow 是 Leg A(EAASP 集成)还是 Leg B(Grid 独立产品)的差异化能力?还是两个 Leg 都该有?**这条直接喂 §E.5 ADR-V2-024 修订决策**。

**与 §B.2 切分表的关系**: 如果"做",governance workflow 引擎应当落在 EAASP 哪一层(L3 / L4 / 跨层)?是新增组件还是 l3-governance / l4-orchestration 扩展?

#### Q3: ⚫ 6 项不在 EAASP 范围的事,EAASP 是不是要为它们留接入位?

L1.5 SCADA-MCP / 国密签名 / WORM 存储 / 极端 case 库 / 信创全栈 / registry 部署 — EAASP 不实现,但 EAASP 是不是要**保证它们装得进来**?

- **答"是"**: EAASP mcp-orchestrator 支持任意第三方 MCP server 装载(**已经支持**);anchors schema 支持外部签名字段(**字段层面已对位**);event 模型兼容外部 audit chain;LLM provider 接口兼容信创 LLM(**已经支持,OpenAICompatProvider**)。**代价小**,主要是把"已经能装"的事**显式文档化**。
- **答"否"**: 客户 / 厂商集成 EAASP 时会撞墙。

**预期答案 = 是**,因为 EAASP 已经做了 ~80%,文档补一下即可。**audit 顺手就能输出**,不是阻塞项。

#### Q4: Grid 和 EAASP 在规范承接上的优先级?

Grid 侧需求清楚(5-7 项 schema / fingerprint / D134 / D136 等,跟 §B.2 工时倾向对位),EAASP 侧 20+ 项要先解 Q1 / Q2:

- **答"Grid 优先"**: 先做 Grid 5-7 项,EAASP 决策做完再启动。**优势**: 跟 §B.2 user 工时倾向对位,且这些改动跟 Phase 2.5 / 3 / 3.5 工作天然融合。
- **答"EAASP 优先"**: 先决定 Q1 / Q2 / Q3 再做 Grid。**优势**: 架构清晰再写代码不返工。**劣势**: 决策可能拖几个 phase。
- **答"并行"**: Grid 5-7 项跟 EAASP 决策正交,可以同时推。**大概率是真实路径**。

**与 §B.3 的关系**: §B.3 隐含 "Grid 必须支撑企业级",所以 Grid 侧的 5-7 项(尤其是 M5 溯源链 + M9 model_lock + M6 evidence chain 字段)**不是规范驱动,是 Grid 自身企业级品质必需** —— 答"Grid 优先" / "并行" 都跟规范脱钩,主动权在 Grid 自己。

### F.2 关键观察(给 audit 用,不是结论)

1. **规范 v0.1 没有让 audit 范围变大,反而让 audit 问题具体化**。§C.1 "engine vs data/integration" 在没有规范评估前是抽象切分,有了 30 项具体清单后,Q1 / Q2 的回答会扎实得多。

2. **Grid 侧的 5-7 项跟规范脱钩** —— 是 Grid 自身演进自然对齐,不是规范驱动。这些事**不需要等 audit 决定**,可以独立推进。

3. **EAASP 侧 13 项 🔴 集中在两个方向(Q1 / Q2),回答不同方向后再做的事不一样**。先答 Q1 / Q2 再做事,不要先做事再发现方向错。

4. **Q3 是 audit 的"顺手便宜事"**,Q4 大概率"并行" —— **真正难的是 Q1 + Q2**。建议 audit 议程把 Q1 / Q2 排在最前。

### F.3 与 §D Sanity Guards 的衔接

§D guards 在 audit 给出颠覆性结论时要求硬证据。F 段引入的 4 个 Q 各自的回答**不会直接触发 §D guards**(因为它们不直接颠覆 §B 切分),但**Q1 答"是"或 Q2 答"做" 会让 §B.2 切分表第 4-6 行(L2/L3/L4 引擎)的工程量大幅膨胀** —— 这要求 audit 显式说明"是否接受 EAASP 引擎层工时倍增是合理的"。

audit guard 候补建议(若 audit 接受时再追加到 §D):
- ⚠️ "Q1 答是 + Q2 答做" 但 §B.2 user 工时倾向 Grid → audit 必须解释 EAASP 引擎层倍增工时由谁承担,user 工时是否要重新分配
- ⚠️ "Q1 / Q2 都答否" → audit 必须解释 EAASP 在企业市场的差异化价值是什么(否则规范 §8 governance 论述会让 EAASP 在企业市场失去竞争力)

> **再强调**: §F 是 agenda 不是 backlog。任何具体平台改动清单必须等 audit 回答 Q1-Q4 之后才能拆。
